# AI 컨텍스트 전략 — 같은 모델, 다른 결과

## 핵심 메시지
AI에게 "뭘 시키느냐"보다 "어떤 상태에서 시키느냐"가 결과를 바꾼다.

---

## 문제: 컨텍스트 오염

코드를 직접 구현한 AI에게 그 코드를 리뷰시키면?

```
구현: "이 방식이 최선이야" (확신)
리뷰: "이 코드는 잘 작성되어 있습니다" (자기 확증 편향)
```

사람도 마찬가지. 내가 짠 코드를 내가 리뷰하면 버그를 못 찾는다.

---

## 해결: context: fork

같은 모델이지만 **구현 맥락 없이** 코드만 보고 리뷰.

```
세션 A (구현): plan → implement (컨텍스트 축적)
세션 B (리뷰): 코드만 읽고 리뷰 (깨끗한 컨텍스트)
```

이걸 **Test-Time Compute**라고 부름.
같은 모델 + 같은 파라미터인데 컨텍스트만 분리하면 결과 품질이 올라간다.

---

## 실제 적용

fork로 실행되는 스킬들:
- **탐색**: explore (탐색 결과가 구현 판단에 영향 주지 않도록)
- **보안**: security-scan (보안은 독립적 시각 필요)
- **계획 리뷰**: plan-ceo-review (구현 가능성에 끌려가지 않고 제품 가치 평가)
- **코드 단순화**: simplify (구현 맥락 없이 객관적으로 복잡도 판단)
- **외부 PR 리뷰**: review-pr (남의 코드는 당연히 별도 컨텍스트)
- **병렬 실행**: batch (독립 work tree에서 실행)

fork 아닌 스킬들:
- **구현**: implement, fix (이전 맥락이 필요)
- **프레임워크 리뷰**: {fw}-03-review (구현 맥락을 알아야 정확한 리뷰)
- **QA**: qa (구현 의도를 알아야 정확한 테스트)

---

## 핵심 원리

| 원칙 | 설명 |
|------|------|
| 구현과 리뷰는 분리 | 같은 사람(AI)이라도 컨텍스트를 끊으면 객관성 확보 |
| 탐색은 오염원 | 코드 탐색 결과가 쌓이면 "이 코드가 맞다"는 편향 생김 |
| 에이전트 분업 | 하나의 AI에게 모든 역할을 주지 않는다 |

---

## 데모 시나리오 (선택)

1. 같은 코드를 같은 세션에서 리뷰 → "잘 되어 있습니다"
2. 같은 코드를 fork 컨텍스트에서 리뷰 → 실제 이슈 발견

---

## 한줄 요약
> AI도 사람처럼 "자기가 한 일"에 편향된다. 컨텍스트를 분리해서 객관성을 확보하라.
